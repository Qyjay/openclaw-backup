### 架构讨论

#### TokenPlan 套餐限制
- Plus-极速版月度套餐：1500 次/月，5 小时计算时间
- 截止 06/03/2026
- 支持 MiniMax-M2.7-highspeed (~100 TPS)
- 音乐生成最耗计算时间（~60-120s/次）
- **结论:** 开发用 Mock，集成测试才开真实 API

#### AI-Native 架构方案（深度讨论）
前辈希望每个用户底层都有一个 OpenClaw agent，实现真正的 AI-Native。

**最终方案：混合架构**
```
确定性操作（CRUD）→ FastAPI 直连 → SQLite（零 LLM 消耗）
AI 操作（对话/生成/推荐/推送）→ FastAPI → OpenClaw Agent → MiniMax API
```

**OpenClaw multi-agent 方案：**
- 一个 Gateway，每个用户一个独立 agent
- 每个 agent 有独立 workspace (SOUL.md + MEMORY.md)
- 注册时自动创建 agent + workspace + 重启 Gateway
- Skill 封装：agent 通过 skill 读后端 API 获取日记数据辅助 AI 决策
- Cron 定时推送（早安/晚安/写日记提醒）
- 支持接入用户自建 OpenClaw（设置页配置 Gateway URL + Token）
- 三层降级：自建 OpenClaw → 内置 OpenClaw agent → 直接 MiniMax API

**A2A 协议调研：**
- Google 发起，Linux Foundation 托管，Apache 2.0
- Python SDK: `a2a-sdk`，原生支持 FastAPI
- 仓库: github.com/a2aproject/A2A
- 前辈最终确认不用 A2A，用 OpenClaw-to-OpenClaw 通信

**扩展性评估：**
- 比赛规模（几十用户）→ 完全可行
- 中等规模（几百用户）→ 需要优化
- 大规模（千+）→ 需要多 Gateway 实例或自建 agent 管理

## 日迹 App v2 大改造（15:00 启动）

### 需求变更（前辈与组员讨论后确定）
砍掉了学习模块（番茄钟/待办），运势功能也降低优先级。聚焦两大核心：

**AI 日记（核心优先级）：**
1. 原始素材收集（拍照+语音+文字，图片位置信息）
2. 素材编辑（图片编辑、AI文字润色【个性化风格标签+提示词】）+ 每条素材情绪提取
3. 每日日记AI生成（日期+天气+图片理解+素材 → 生成日记）+ 用户可修改（次数限制）+ 情绪趋势图
4. 日记总结 → 漫画/小说/视频 + 分享
5. 日记分享卡片（用户选择分享范围）
6. 纪念日 + 那年今日（手动添加 + AI 自动感知）
7. 日记信息提取（偏好、人物关系、纪念日）
8. 主动推送（天气、时光胶囊、纪念日）

**社交（A2A 框架）：**
1. 用户画像（个人资料+AI聊天+日记数据）
2. Agent 间交流
3. 匹配（长期交流+匹配报告+双向选择）
4. 短期搭子（申请-同意）
5. 功能封装（通过AI对话完成所有功能）

### 关键架构决策

#### AI 功能分层（重要）
- **确定性AI功能 → 直接调 MiniMax API**（快、准、省 token）
  - 情绪提取、文字润色、日记生成、漫画/小说、信息提取、用户画像
  - 固定 prompt 模板 + JSON mode 确定性输出
  - 延迟 1-3 秒，每次 1 次 API 调用
- **对话+推送 → 走 OpenClaw Agent**（需要记忆和上下文）
  - AI 助手对话（SSE）、主动推送（cron）、Agent 间交流
  - 延迟可接受（对话本身有等待预期）
- **原因：走 OpenClaw skill 每次需要 3 次 LLM 推理（理解意图+执行+组织回复），延迟 10-30s，输出是自然语言不是结构化 JSON，token 消耗 3-4 倍**

#### 个性化方案
- 直接调 API 时：从数据库读用户画像/风格标签 → 注入 system prompt
- 走 OpenClaw 时：MEMORY.md 积累对话记忆
