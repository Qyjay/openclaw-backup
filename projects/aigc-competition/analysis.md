## 二、vivo 技术生态盘点

### 2.1 蓝心大模型 API（比赛免费提供）

**接口地址：** `https://api-ai.vivo.com.cn/v1/chat/completions`（OpenAI 兼容协议）

| 能力 | 可用模型/说明 | 对产品的价值 |
|---|---|---|
| **文本生成** | DeepSeek-V3.2, Doubao-Seed-2.0 (mini/lite/pro), Qwen3.5-plus | 日记文案生成核心引擎 |
| **图片理解（Vision）** | DeepSeek-V3.2, Doubao-Seed-2.0 系列 | ⭐ **核心能力**：照片→理解→日记，替代用户手写 |
| **Function Calling** | 全模型支持 | Agent 工具调用，智能体行为引擎 |
| **深度思考（Reasoning）** | Doubao-Seed-2.0 默认开启 | 复杂推理场景（用户画像分析、匹配逻辑） |
| **AI 绘画** | 独立接口 | 日记配图美化、风格化处理 |
| **ASR（语音识别）** | 独立接口 | 语音输入→日记文字 |
| **TTS（语音合成）** | 独立接口 | 日记朗读、语音陪伴 |
| **LBS（位置服务）** | 独立接口 | 自动记录地点，丰富日记上下文 |

### 2.2 其他平台工具

- **快应用开放平台** — 轻量级应用形态，无需安装，vivo 系统原生支持
- **蓝心九问** — vivo 自带的 AI 问答助手
- **蓝心智绘** — vivo 自带的 AI 绘画工具

### 2.3 端侧大模型（复赛开放）

复赛阶段 vivo 将开放蓝心端侧模型。**这是一个重要的差异化机会**：
- 端侧推理 = 无网络依赖 + 低延迟 + 隐私保护
- 如果初赛就为端侧化设计好架构，复赛可以快速适配
- **建议：初赛用云端 API，但架构上预留端侧替换接口**

## 三、竞品分析

### 3.1 日记类

| 产品 | 特点 | 局限 |
|---|---|---|
| Day One | 精美 UI、多媒体、地图时间线 | 纯手动记录，无 AI 生成 |
| 格志日记 | 问答式引导记录 | 模板化，缺乏个性 |
| Notion / Flomo | 笔记工具，可做日记 | 不是专门日记产品，无 AI 理解 |
| 心暖日记 | 情绪追踪 + 简单记录 | 功能单一 |

**空白点：** 没有一款主流日记 App 能做到"拍照即生成日记"——这是真正的零门槛。

### 3.2 AI 社交类

| 产品 | 特点 | 局限 |
|---|---|---|
| Soul | 匿名社交，兴趣匹配 | 人工填标签，非 AI 驱动 |
| 小红书 | 内容社交 | 不是精准匹配 |
| 探探/Bumble | 左右滑动匹配 | 基于照片，浅层匹配 |
| Replika | AI 陪伴 | 单人与 AI 互动，非 A2A |

**空白点：** 没有产品做到"基于用户真实生活数据构建数字分身 → 分身代理社交"。A2A 概念在市面上基本没有落地产品。

### 3.3 差异化总结

你们的产品方向确实是蓝海：**日记 × AI 生成 × 数字分身 × 社交**，这个交叉赛道几乎没有直接竞品。关键是如何在比赛时间内**做出来**。

## 五、初赛 MVP 定义

### 5.1 核心原则

**砍到骨头，但让骨架完整。** 评委需要看到：
1. 一个能跑在 vivo 手机上的可用 demo
2. 大模型能力的深度运用（不是简单调 API）
3. 产品闭环（至少能演示完整链路）
4. 创新性（别人没做过的东西）

### 5.2 MVP 功能清单（初赛必做）

#### ✅ P0 — 必须交付

| 功能 | 说明 | 技术方案 |
|---|---|---|
| **拍照/上传图片** | 支持选择相册照片或即时拍照 | Android Camera/Gallery API |
| **AI 日记生成** | 上传 1~N 张照片 → 自动生成日记文字 | **蓝心 Vision API**：图片理解 + 文本生成，结合时间/地点上下文 |
| **日记浏览/时间线** | 按日期查看历史日记 | 本地 SQLite + RecyclerView |
| **AI 风格化** | 日记文风可选（文艺/幽默/简洁/治愈） | System prompt 切换 |
| **个人知识库** | 所有日记数据向量化存储，支持语义检索 | 本地 embedding + 简单向量检索（或云端 RAG） |
| **智能体画像** | 基于日记自动提取用户兴趣标签 | Function Calling：定期分析日记，输出结构化标签 |

#### 🔶 P1 — 强烈建议（加分项）

| 功能 | 说明 | 技术方案 |
|---|---|---|
| **语音输入** | 说一段话 → 转文字 → 融入日记 | 蓝心 ASR API |
| **AI 配图美化** | 对照片进行风格化处理 | 蓝心 AI 绘画 API（图生图） |
| **每周/每月 AI 总结** | 一键生成生活复盘报告 | RAG 检索近期日记 + 文本生成 |
| **智能体对话** | 和自己的数字分身聊天 | RAG + Chat API，让 AI 以"你"的口吻回答 |

#### ❌ P2 — 初赛不做（复赛/决赛迭代）

| 功能 | 原因 |
|---|---|
| A2A 智能体社交 | 需要多用户、后端基建，7 周做不完 |
| 后台无感社交匹配 | 冷启动问题，demo 无法有效展示 |
| AI 音乐/视频生成 | 锦上添花但不影响核心价值，分散精力 |
| 图片手绘编辑 | 技术门槛高，ROI 低 |
| 智能设备数据接入 | 依赖外部硬件，不可控 |

### 5.3 产品形态建议

**推荐：Android 原生 APP**（而非快应用或 H5）
- 理由：原生应用完成度更高、体验更好、可调用系统能力（相机/相册/通知）
- 为复赛端侧模型适配做准备
- 技术栈：Kotlin + Jetpack Compose（现代 Android 开发）
- 后端：轻量 Python/Node.js 服务器 + 蓝心 API 转发

## 六、大模型应用亮点设计

这是评审维度里权重很高的部分，需要做出"技术深度"和"用法创新"。

### 亮点一：多模态日记引擎（Vision + NLG 深度融合）

**不是简单的"看图说话"，而是：**

```
[照片] + [时间] + [位置] + [天气] + [历史日记风格] + [用户人设]
              ↓ 融合推理
      [贴合用户语气的个性化日记]
```

技术实现：
1. Vision API 提取图片内容（场景、人物、物品、情绪）
2. 系统 prompt 注入上下文（时间、地点、天气、用户之前的写作风格）
3. Function Calling 调用 LBS 获取位置、调用天气 API
4. RAG 检索用户历史日记，学习语气和关注点
5. **让生成的日记"像用户自己写的"，而非通用 AI 输出**

→ 这比单纯调 Vision API 看图写一段话要深入得多。

### 亮点二：渐进式智能体构建（数据飞轮）

**智能体不是一次性生成的，而是随着使用不断进化的。**

```
Day 1: 用户画像 = [空白]
Day 7: 用户画像 = {喜欢咖啡, 常去图书馆, 语气偏文艺}
Day 30: 用户画像 = {咖啡爱好者(偏手冲), 周末图书馆常客, 文艺但偶尔吐槽, 
                    最近在准备雅思, 喜欢拍猫...}
```

技术实现：
1. 每次日记生成后，后台 Function Calling 自动提取结构化标签
2. 标签存入用户画像知识库（向量化）
3. 画像反哺日记生成（越用越懂你）
4. **可视化展示画像成长过程**——这在 demo 演示时非常有效

### 亮点三：AI 工具链编排（Function Calling 深度应用）

展示大模型不只是"文本生成器"，而是**智能调度中心**：

```
用户上传一张照片
  → LLM 判断：这是一张旅行照
  → 自动调用：LBS（获取地点）+ Weather API（当时天气）+ Vision（场景分析）
  → 综合生成：带地点、天气、情绪的完整日记
  → 自动分类：打上"旅行"标签
  → 自动推荐：是否生成本周旅行合集？
```

## 七、冷启动策略

### 7.1 初赛阶段（单人可体验）

初赛 demo 不需要"社交"，聚焦**单用户的日记 + 智能体构建**即可。
- 评委下载 APP → 拍几张照 → 看到 AI 日记 → 看到智能体画像 → 和自己的分身对话
- **这个体验链路完全不依赖其他用户**

### 7.2 社交功能冷启动（复赛/决赛讲故事）

如果进入复赛需要展示社交能力：

**方案 A：团队成员互测**
- 5 个队员 + 指导老师 + 若干同学注册使用
- 积累一定日记数据后，在小范围内测试 A2A 匹配

**方案 B：模拟演示**
- 预置 10-20 个虚拟用户画像（基于真实场景构造）
- 演示智能体如何进行后台匹配和推荐
- 展示匹配逻辑和结果，而非依赖真实用户量

**方案 C：校园场景切入（如果有时间做）**
- 限定场景：南开校内"找自习搭子""找饭搭子"
- 先在校内小范围推广获取种子用户
- 校园场景天然有地理聚集性，降低冷启动难度

### 7.3 初赛 PPT/演示中如何呈现社交

社交是你们的**核心差异化**，不能完全不提。建议：
- 初赛做日记 + 智能体 MVP → 实际可用
- 社交部分用 **交互原型 + 架构图 + 技术方案** 呈现
- 重点讲"A2A 如何工作"的技术逻辑，而非呈现空壳功能
- 在答辩/文档中画出清晰的**未来路线图**

## 八、vivo 生态适配方案

### 8.1 初赛

| 适配点 | 方案 |
|---|---|
| **运行环境** | 标准 Android APK，兼容 vivo 手机 Android 13+ |
| **API 调用** | 全部使用蓝心大模型 API（OpenAI 兼容，对接简单） |
| **系统能力** | 调用 vivo 手机相机、相册、通知（标准 Android API） |

### 8.2 复赛（蓝心端侧模型适配）

| 适配点 | 方案 |
|---|---|
| **端侧推理** | 日记风格化、标签提取等轻量任务 → 端侧模型 |
| **云端推理** | 复杂多图日记生成、RAG 检索 → 云端 API |
| **混合架构** | 设计统一推理接口，初赛云端实现，复赛可切换为端侧 |
| **隐私优势** | 端侧处理个人数据，无需上传服务器 — 社交场景的信任基础 |

## 十、风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| Vision API 生成质量不稳定 | 日记质量不可控 | 多模型 fallback + 后处理优化 prompt |
| 7 周时间不够 | 完成度低 | 严格砍 P2，保 P0 |
| vivo 手机适配问题 | 无法演示 | 提前借一台 vivo 手机测试 |
| 社交功能没做出来 | 评审觉得不完整 | 初赛聚焦日记+智能体，社交用架构图讲 |
| API 调用限流/配额 | 开发受阻 | 开发阶段用 mock 数据 + 限制真实调用 |

## 十一、结论与建议

### 产品方向：✅ 认可

日记 + AI 生成 + 数字分身这个方向是成立的，竞品少，切入点清晰。

### 关键调整

1. **初赛砍掉社交功能的实现**，聚焦"AI 日记 + 智能体构建"做到能用、好用
2. **社交作为核心差异化在文档/答辩中重点阐述**，展示技术方案和未来路线
3. **大模型应用要做深**，不是调 API，而是 Vision + Function Calling + RAG + 个性化的深度融合
4. **为复赛端侧化提前设计架构**，展示技术前瞻性
5. **尽快借一台 vivo 手机**，真机调试
