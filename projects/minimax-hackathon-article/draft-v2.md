# MiniMax 黑客松 · 开发者故事
## 当 AI 走出屏幕：这群开发者做了什么

---

凌晨两点，一间亮着灯的会议室里，有人盯着屏幕上的报错信息发呆，有人揉着眼睛把同一行代码跑了第八遍，有人靠在椅背上假装休息实际上睡不着。

这是每一场黑客松都会出现的场景。

过去几个月，MiniMax 赞助了多场高校和开发者社区的黑客松。我们提供比赛期间的 API 调用额度，让参赛者不用为 token 预算发愁；我们派出技术团队的同事担任评委和嘉宾，分享在实际产品里用模型的经验；也给获奖团队准备了 CodingPlan，支持他们在赛后继续把项目跑下去。

做这些事情的理由很简单：我们想看看，当开发者拿到一个足够好用的模型之后，会用它做出什么东西来。

答案比我们预期的更有意思。

这篇文章想分享其中七个项目。它们有的已经在比赛结束后继续迭代，有的还在想办法走得更远，有的只是想清楚了接下来要做什么。我们没有按「创新程度」或者「商业潜力」来排序——它们各自解决不同的问题，有不同的完成度，也面对不同的挑战放在一起看，只是因为它们加起来，让我们看到了某种正在发生的变化。

## 01 虾虾侦探BP：别只做屏幕里的 AI

大多数 AI 应用还在屏幕里卷。

聊天窗口、对话框、生成式搜索——这些都是屏幕里的 AI。它让大模型的能力变成了文字，再把文字交给用户。是实用的，也是有限制的。

虾虾侦探BP 想做的事，是把这个限制打破。

它是一个 AI 驱动的实体剧本杀游戏。AI 扮演 DM（游戏主持人），根据剧情推进生成对话和指令，同时控制一条真实的机械臂在桌面上操作道具。玩家面对的不是一个弹窗，而是一个有物理存在感的东西——AI 角色会真的伸手去推动剧情里的一个物件。

这套系统的技术底座是 OpenClaw 的 Gateway+Node 架构，AI 部分则由 MiniMax 的模型承担。团队需要模型做的事比较复杂：理解当前游戏状态、判断剧情走向、生成符合角色设定的对白，同时输出控制机械臂的指令序列。上下文理解要够稳，指令跟随要够准，不然机械臂就会在错误的时机做错误的事。

技术方案是项目的基础，但让我们印象更深的，是他们路演前那一夜。

那天下午，机械臂开始出问题。每次完成一个动作之后，它会莫名其妙地死机。团队 debug 到晚上十一二点，连一个能稳定运行的 demo 都还没有。另一边，OpenClaw 的剧本输出顺序也怎么调都不对。路演在第二天中午，团队几乎觉得自己要交白卷了。

开发者的原话是：「又困又累又崩溃，感觉什么都交不出来了。但抬头看到队友们还在那么认真地为这个项目拼，脑子里就冒出一个声音——要不再试试？再坚持一下吧。」

凌晨一点多，机械臂终于稳定了。几乎是同一刻，剧本输出也调通了。他形容那个瞬间「像是电影里主角团绝处逢生的经典桥段」。

这件事让我们想留下一句话：**大模型能力已经足够强了，现在最缺的不是更聪明的模型，而是更有想象力的交互方式。机械臂、灯光、音箱，这些东西接入 AI 的门槛比你想象的低得多。**

去试试。

## 02 Mira：有些关心，不需要开口

用过 Replika、Pi、Character.AI 吗？

Mira 团队是这些产品的深度用户。用了一阵子之后，他们发现了一个共同的问题：这些产品都在等你开口。你不说，它就不动。你不提问，它就沉默。

但真正的关心不是这样的。

小狗看到你难过就会凑过来，妈妈看你放学回家脸色不对会先问「怎么了」，不会等你先开口。这是人类世界里最基本的照顾逻辑——**先看到，再判断，然后做点什么**。但现有的 AI 伴侣都建立在一个相反的前提上：你要先表达，我才回应。

Mira 想做的，就是把这个逻辑反过来。

它是一个主动感知型的 AI 伴侣，底层框架基于 OpenClaw 构建，模型能力由 MiniMax M2.5-highspeed 提供。具体的工作方式是：借助 AR 眼镜、可穿戴信号和环境上下文，实时判断用户状态，然后在用户最不愿意主动开口的时刻——而不是最需要说话的时刻——给出回应。

这个闭环分四层：主动感知、记忆理解、分判断、设备执行。AR 眼镜看你在什么场景，手环知道你心跳和体征变化，AI 判断需不需要介入，如果判断需要，就联动智能家居执行动作。整个过程不需要用户说一个字。

这个方向背后有一组被团队反复提及的数字：中国有 1.3 亿独居老人，900 万留守儿童，上亿独居青年。这些人不会主动说「我今天不舒服」或者「我今天很累」。现有的 AI 伴侣对他们来说是无效的——因为它们需要开口作为触发条件，而这些人恰恰是最不会主动开口的群体。

在 AttraX 黑客松的现场，团队经历了典型的 24 小时地狱模式。周日凌晨三点之前，打印机、智能眼镜、智能手环、手机，四个硬件端没有一个真正跑通。整个项目最关键的「感知—判断—执行」闭环，卡在最后几步落不下来。Thomas 一个人扛了那几个小时，接协议、查状态、修触发、补桥接，等到日出的时候，四个硬件端全部打通了。那一刻 Mira 才第一次不只是想法，而是一个真正跑起来的系统。

但让团队确定自己走对路的，是一个更小的瞬间。

路演前，创始人戴着 Rokid 眼镜站在后台。Mira 通过眼镜看到台下坐满了人，通过手表感知到他心跳加速，调取了这几天准备的记忆，然后在他耳边说：「放轻松，你肯定可以做到。这两天你准备了很多，去拿下这个舞台吧。」

他没有按任何按钮，没有说「我很紧张」。那一刻他知道，这个方向是对的。

他们想把这句话留给其他开发者：**有些关心，不需要开口。只要你有想法，那就赶紧把它 build 出来。**

## 03 AIHost：让每一场活动都有一位永不离场的主持人

活动主持是一件被严重低估成本的事。

赛前写稿，半天到一天。赛中连续高强度工作数小时，任何临时变动都要靠主持人现场扛。台上一个人，脑子里装着所有流程，嘴里要实时产出过渡语，眼睛要观察台下反应——这些动作同时发生，全靠一个人撑着。

AIHost 想解决的问题很简单：这件事，为什么需要人来做？

它是一个面向线上和线下活动场景的 AI 主持人产品，覆盖从主持稿生成、播报音频生成到现场自动口播与实时观众互动的全流程。核心有三个功能：主持稿智能生成（上传议程图片，多模态模型理解流程，自动生成完整主持稿）；AI 驱动的动态口播（在嘉宾演讲过程中，系统持续记录 ASR 转录内容，引用嘉宾实际发言生成个性化过渡词，而不是固定台词）；自动主持（流式 ASR 实时监听，自动触发口播）。

团队只有一个人。刘海丰，独立开发者，产品设计、全栈开发、现场测试一人包揽。他是那场 OpenClaw 黑客松里全场唯二的 solo 参赛者。

从周六晚上到周日早上，他连续通宵了大约 8 到 10 个小时。中午产品上线，下午就直接上台——用 AIHost 主持自己的路演环节。

他说，路演的那一刻，他把现场主持人静音，切换到 AIHost，让它来播报「接下来是刘海丰同学的 AIHost 项目」。评委提问结束之后，AIHost 自动播报「感谢各位评委老师的提问」。

那一刻产品不再只是 demo，而是在替代一件现实中一直需要人做的事。

目前 AIHost 已经接入了 MiniMax 的语音合成与多模态理解能力。TTS 用来生成更有表现力的主持音色，多模态用来理解复杂议程图片（比如多栏布局或者时间轴格式）。后续计划把音色选择页面和稿件生成环节都跑在 MiniMax 的能力上。

刘海丰对其他开发者说的一句话是：**做产品最难的事之一，是一直找不到 PMF、没有反馈、不知道自己到底解决了什么问题。AIHost 的第一次验证不是在测试环境里，而是在真实的 200 人活动现场。那次之后，意向订单来了，用户反馈来了，迭代方向也清晰了。先让产品真正落地。**

## 04 OpenDiary：被看见的感觉

你有没有过这种感觉——

写日记，写着写着就停了。不是因为懒，而是因为写了也没人看。三个月前写下的一句话，后来再也没有被翻出来过。那些思考、情绪、决定，就这样散落在不同的 app 里，没有一个地方能把它们串起来。

OpenDiary 想解决的问题，就是这个。

它是一个基于 MiniMax 大模型的 OpenClaw Skill，核心理念是让记录成为对话的自然延伸。用户不需要打开一个专门的 app，不需要找到「写日记」那个入口，直接用自然语言跟 OpenClaw 说就行。AI 帮你整理、提炼、关联，把碎片化的内容变成有结构的东西。

这个过程中最有价值的部分，不是「记录」本身，而是「被记录之后发生的事情」。

团队第一次跑通 Note Extractor（笔记洞察提取功能）的时候，看到 AI 从一堆碎片日记里提炼出了「你这个月最在意的三件事」。那一刻他们意识到，这个东西是真的有用的。

不是帮你写得更整齐，而是帮你看见自己。

OpenDiary 还有一个有意思的设计：双人设模式。INTJ 模式下 AI 更理性，帮你整理逻辑和事实；ENFP 模式下更活泼，帮你捕捉情绪和灵感。不同的对话人格，让记录这件事有不同的温度。

记录分散在不同地方这个问题，也被他们考虑到了。系统支持从微信、飞书、Flomo、Obsidian、Notion 等已有工具导入内容——不需要改变现有的工作习惯，AI 在背后帮你把碎片拼起来。

团队成员 Vivi 说过一句话：**「AI 记录让我们过的每一天都作数。」** 当 AI 能够把你三个月前说的一句话和今天的决定联系起来，那种感觉才是真正的第二大脑。

他们想对开发者说的话是：**不要等到「准备好了」再开始记录。AI 时代最大的红利，属于那些有记录的人。不要怕没技术背景，这个时代 idea / 执行力 / 商业化更重要了。开始享受玩 AI 吧。**

## 05 RemoteLab：让 AI 替普通人工作

RemoteLab 的创始人酒嘉年，之前在字节跳动的 CloudIDE 团队和 TikTok 内容安全团队做全栈开发。他不是在互联网行业观察 AI，他是在 AI 行业做互联网产品。这种交叉视角让他看到了一个问题。

很多 Agent 产品，默认用户已经很懂 AI：知道怎么写 prompt，知道怎么调参数，知道怎么设计工作流。但真正有大量重复数字工作的人——那些每天要做表格整理、文件处理、信息归纳、导入导出的人——往往不具备这些前提。他们面对一个空白输入框，根本不知道该说什么。

RemoteLab 想做的，就是把门槛降下来。

它的定位是一个「自己拥有电脑的个人 AI 数字助理」。用户不需要先想清楚怎么描述任务，把文件、截图或者一个模糊的目标扔进来就行。AI 会先帮你梳理需求：你到底要做什么？输入是什么？输出应该长什么样？搞清楚之后，AI 在一台真实的电脑上开始执行，不需要你在屏幕前盯着。支持手机和桌面端接入，任务中断了也不会丢上下文，下次打开接着来。

有几个具体的设计细节：不是假设用户一开始就能清楚表达需求，而是让 AI 主动帮助用户澄清任务；不是只停留在聊天层，而是让 AI 在真实电脑上持续完成任务；通过持久化 session 和记忆机制，让任务不会因为中断就丢失上下文。

酒嘉年在接受采访的时候说的一句话，我们觉得是今年听到的比较准确的一个判断：**「问题不是模型不够强，而是人根本不知道怎么开始。谁能把入口做得更简单、更自然、更可引导，谁就更有机会把 Agent 带给更大的人群。」**

他还说：**「Agent 的普及，未必先发生在最懂技术的人群里，反而可能先发生在那些有大量真实数字工作、但过去一直进不来的人群里。」**

RemoteLab 已经开源，有兴趣的开发者可以在文末找到 GitHub 链接和体验地址。

## 06 ClawSight：让普通人也能拥有 Bloomberg 终端

金融市场上每天都在发生大量的信息流动：新闻、公告、数据波动、社交媒体讨论、政策文件……专业机构有专门的研究团队来做信息处理和判断，但普通人面对这些信息噪音，往往很难判断哪条重要、哪些只是噪音、这件事会不会继续发酵、会影响哪些方向。

ClawSight 想做的，是把这个能力压缩进一个个人人都能用的终端里。

它的定位是「个人 Bloomberg 终端」，核心技术是多 Agent 协作。具体做法是把复杂的研究任务拆成多个角色：信息采集 Agent 负责抓取和筛选，事实验证 Agent 负责核查来源和交叉比对，事件框架 Agent 负责组织逻辑，历史案例 Agent 负责寻找类比，市场影响 Agent 负责分析传导路径——最后由主 Agent 汇总输出。

这种分工的好处是：每个 Agent 只做自己最擅长的事，输出结果更稳定，也更容易持续优化。系统不是一次性回答问题，而是在运行中不断沉淀判断样本，让自己的能力越来越像一个真正会成长的研究系统。

ClawSight 基于 MiniMax M2.5 大模型搭建，核心用到了两个能力：大参数量级推理能力（复杂金融任务处理稳定，逻辑连贯闭环），以及全量历史信息承载与长上下文适配（金融市场历史数据与舆情储备完整，长流程分析不丢关键信息）。具体落地了两项功能：多维度信息核验筛选（依托完备历史信息储备，交叉核验信源真伪、过滤市场噪音）和市场事件传导分析（凭借高效推理能力，深度推演事件影响链路与市场传导逻辑）。

团队印象最深的一个瞬间，是多个 Agent 串起来之后，系统不只是在把新闻重复转述，而是真的开始自动拆分问题、补充上下文、组织事件逻辑、给出更完整的判断。那一刻他们觉得，这不只是一个 Demo，而是一个真正有机会变成可用的产品。

他们说：**「最难的不是让模型『会说』，而是让系统『会工作』。当 Agent 真正开始像一个个职能明确的小团队协作时，整个产品的质感会完全不一样。」**

## 07 数字生命卡

数字生命卡是这次分享中最难定义的一个项目。

它的基本理念是：把人的记忆、知识和经验结构化地存储下来，生成一个可以被持续交互的数字版本。这个方向在技术上涉及多模态内容的处理和生成，在应用场景上则指向某种更长期的个人数据资产管理。

我们没有拿到这个项目完整的开发者故事素材，但它所在的赛道方向本身值得关注：如何在 AI 时代让个人的知识和记忆真正成为可被调用、可被延续的东西，这件事还没有标准答案。感兴趣的开发者可以访问 ModelScope 页面了解更多信息。

## 从这七个项目里，我们看到了什么

把这七个项目放在一起看，有几件事是明确的。

第一，开发者对 AI 的想象力正在脱离屏幕。虾虾侦探让 AI 操作真实的物理动作，Mira 让 AI 在你没说话的时候主动介入，AIHost 让 AI 控制真实场景中的语音输出——这些项目都在把 AI 从一个「生成文字的工具」变成一个「与真实世界产生关系的系统」。这不是噱头，这是真实的产品思路在拓展。

第二，「让普通人用得上」正在从一句口号变成真实的产品命题。RemoteLab 直接面向不懂技术的用户设计交互，OpenDiary 要解决的是每个人都在经历但没有人解决的碎片化记录问题，Mira 服务的对象是那些最需要被照顾但最不会主动求助的人。这些项目背后的判断是：AI 的下一阶段，不只是更强，而是更可及。

第三，AI 应用的产品形态远比我们想象的更分散。从伴侣到助理，从主持到日记，从 IDE 到金融终端——没有一个统一的模板告诉你 AI 应用应该长什么样。这种多样性本身说明了一件事：我们还处在 AI 应用的非常早期，最好的产品形态还没有被发明出来。

第四，黑客松依然是一个出好东西的地方。有限的时间、极高的压力，逼着团队在极短周期内做出核心决策——用什么架构、解决什么问题、放弃什么功能。那些在舒适区里想三个月都定不下来的事情，在比赛现场反而快速清晰了。

## MiniMax 扮演了什么角色

我们不太想把这部分写成产品介绍。

我们做的事情很简单：比赛期间给参赛团队提供 MiniMax API 调用额度，让他们不用在调试产品的时候还要担心 token 够不够用；派技术团队的同事去线下做评委和嘉宾，分享在实际产品里用模型的一些真实经验和踩过的坑；给获奖团队发 CodingPlan，让他们赛后还有资源继续迭代。

这些项目里有不少直接用 MiniMax 的模型处理核心任务——对话生成、内容理解、任务规划、指令跟随。开发者选择一个模型，通常不是因为品牌，而是因为在他的具体场景里，这个模型的效果、速度和成本之间的平衡最合适。我们希望在这些项目里，MiniMax 扮演的角色就是这样：一个被选中的、好用的技术底座。

同时，这些项目也给我们的模型迭代提供了有价值的反馈。开发者在极限时间压力下使用模型的方式，往往比我们自己设想的更有创意。这些具体的、带着真实场景的反馈，比任何 benchmark 数据都更有参考价值。

有些项目在比赛结束后没有继续更新，它们完成了自己的使命。有些还在持续迭代，每个月都有新功能。有些已经有了真实用户，在真实场景里被使用着。这些都是好的结果。

我们做黑客松赞助这件事，最初的想法没有多复杂：就是想看看，开发者拿到一个足够好用的模型之后，会做出什么东西来。

现在我们知道了——他们做出的东西，比我们预期的更有意思。
