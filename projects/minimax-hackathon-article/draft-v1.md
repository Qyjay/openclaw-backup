## 正文

凌晨一点，一间灯火通明的会议室里突然爆发出一阵欢呼。

一条机械臂在桌面上缓缓抬起，精准地完成了它被设定好的动作——夹起一只小龙虾模型，放到指定位置。这个动作本身不复杂，但在那个时刻，它意味着一切。因为就在十分钟前，整个团队还觉得自己什么都交不出来。

这是「虾虾侦探BP」团队在黑客松路演前一夜的真实经历。也是过去几个月里，我们在所赞助的多场黑客松中，看到的无数个类似瞬间中的一个。

MiniMax 从去年开始赞助了多场高校和开发者社区的黑客松活动，为参赛团队提供比赛期间的 API 调用额度，派出评委和技术嘉宾做分享，并以 CodingPlan 作为奖品支持获奖团队在赛后继续迭代。我们见证了太多从零开始、24 到 48 小时内把一个想法变成可运行产品的故事。这篇文章想做的事情很简单：把其中一些让我们真正觉得「这个有意思」的项目，连同它们背后的开发者故事，分享给更多人。

### 虾虾侦探BP：给 AI 装上一条机械臂

「虾虾侦探BP」是我们在这批项目里第一个想聊的，因为它做了一件大多数 AI 项目不会做的事——走出屏幕。

它是一个 AI 驱动的实体剧本杀游戏。AI 扮演 DM（游戏主持人），根据剧情推进生成对话和指令，同时控制一条真实的机械臂在桌面上操作道具。玩家看到的不是一个聊天窗口，而是一个会动的、有实体存在感的 AI 角色。

整套系统的技术栈建立在 OpenClaw 的 Gateway+Node 架构上。团队在介绍中提到，这个架构让他们不用自己造轮子，把精力集中在产品体验本身。MiniMax 的模型在这里承担了剧本推理和对话生成的工作——AI 需要理解当前的游戏状态，判断剧情走向，生成符合角色设定的对白，同时输出控制机械臂的指令序列。这对模型的上下文理解和指令跟随能力都有相当高的要求。

但比技术方案更让我们印象深刻的，是他们路演前那一夜发生的事。

那天下午开始，机械臂就反复出问题。完成一个动作之后莫名其妙地死机，debug 一直持续到深夜。另一边，OpenClaw 的剧本输出顺序也怎么调都不对。到了晚上十一二点，团队连一个能稳定运行的 demo 都没有，而第二天中午就要上台路演。

开发者后来回忆那个时刻：「又困又累又崩溃，感觉我们什么都交不出来了。但抬头看到队友们还在那么认真地为这个项目拼，脑子里就冒出一个声音——要不再试试？再坚持一下吧。」

凌晨一点多，机械臂终于稳定了。几乎是同一时刻，另一边的剧本输出也调通了。他形容那个瞬间「像是电影里主角团绝处逢生的经典桥段」。

这个团队在赛后留下了三句话，我们觉得值得原文放在这里：

「别只做屏幕里的 AI。大模型能力已经足够强了，现在最缺的不是更聪明的模型，而是更有想象力的交互方式。机械臂、灯光、音箱，这些东西接入 AI 的门槛比你想象的低得多。」

「先跑通再说。我们一天跑通了全链路，不是因为技术多厉害，而是因为选对了架构。好的工具能让你跑得更快。」

「去比赛。不是为了奖金（虽然奖金真的很香），而是因为比赛会逼你在极短时间内把想法变成现实。那种压力下做出来的东西，往往比你在舒适区里打磨三个月的更有生命力。」

### RemoteLab：让 AI 自己拥有一台电脑

如果说虾虾侦探代表的是「AI 走进物理世界」，那 RemoteLab 代表的是另一个同样重要的方向——让 AI 替普通人做事。

RemoteLab 的定位是一个「自己拥有电脑的个人 AI 数字助理」。听起来有点抽象，但它要解决的问题其实很具体：你每天花在表格整理、文件处理、信息归纳、数据导入导出上的时间，能不能交给 AI 来做？

现有的很多 Agent 产品其实已经在尝试解决这个问题了，但 RemoteLab 的创始人酒嘉年——前字节跳动 CloudIDE 团队的全栈工程师——观察到一个被忽视的断层：大部分 Agent 工具默认用户已经很懂 AI、很会写 prompt、很愿意折腾配置。但真正有大量重复数字工作的人，往往不具备这些前提。他们面对一个空白输入框，根本不知道该怎么开始。

RemoteLab 的做法是把这个顺序反过来。用户不需要先想好怎么描述任务。把文件、截图或者一个模糊的目标扔进来就行，AI 会先帮你梳理需求——你到底要做什么？输入是什么？输出应该长什么样？搞清楚之后，AI 在一台真实的电脑上开始执行。支持手机和桌面端接入，任务中断了也不会丢上下文，下次打开接着来。

酒嘉年在采访中说了一句让我们印象很深的话：「问题不是模型不够强，而是人根本不知道怎么开始。谁能把这个入口做得更简单、更自然、更可引导，谁就更有机会把 Agent 带给更大的人群。」

他还有一个判断我们觉得挺有意思：「Agent 的普及，未必先发生在最懂技术的人群里，反而可能先发生在那些有大量真实数字工作、但过去一直进不来的人群里。」

RemoteLab 目前已经开源，有兴趣的开发者可以在文末找到 GitHub 链接和体验地址。

### 从这些项目里，我们看到了什么

把这八个项目放在一起看，有几件事情挺明显的。

第一，开发者对 AI 的想象力正在脱离屏幕。虾虾侦探接了机械臂，其他项目也在各自的方向上尝试让 AI 跟真实世界产生更直接的联系。聊天窗口当然还是最基础的交互形式，但越来越多人已经不满足于此了。

第二，「让普通人用得上」正在成为一个真实的产品命题。RemoteLab 明确提出了这个方向，我们在其他几个项目中也看到了类似的思考。模型能力本身已经不是最大的瓶颈，怎么让不懂技术的人也能上手，怎么把「第一次使用」的门槛降下来——这些问题反而更紧迫。

第三，AI 原生应用的形态比我们想象的更多样。从伴侣到助理，从日记到 IDE，从剧本杀到数字生命——没有一个统一的模板告诉你「AI 应用应该长什么样」。这种多样性本身就说明了一件事：我们还处在 AI 应用的非常早期，最好的产品形态还没有被发明出来。

第四，也是我们作为赞助方最直观的感受——黑客松真的是一个出好东西的地方。有限的时间、极高的压力，反而逼出了很多在舒适环境下做不出来的产品决策和技术选型。好几个项目在比赛结束后都选择了继续迭代，这对我们来说是最好的结果。

### MiniMax 在这些故事里扮演了什么角色

说实话，我们不太想把这部分写成一段产品介绍。我们做的事情很直接：比赛期间给所有参赛团队提供 MiniMax 的 API 调用额度，让大家不用为 token 预算发愁；派技术团队的同事去做评委和嘉宾分享，聊聊我们在模型能力和应用层面的一些经验和观察；给获奖团队发 CodingPlan，让好项目在赛后还能继续跑下去。

从结果来看，这些项目中有不少直接使用了 MiniMax 的模型来处理核心任务——对话生成、内容理解、任务规划、指令跟随。开发者选择一个模型，通常不是因为品牌，而是因为在他们的具体场景里，这个模型的效果、速度和成本之间的平衡最合适。我们希望 MiniMax 在这些项目里扮演的角色就是这样：一个被选中的、好用的技术底座。不需要额外的故事包装。

我们也在这些项目中学到了不少东西。开发者在极限时间内对模型的使用方式，往往比我们自己设想的更有创意。这些反馈对我们自己做模型优化和 API 设计也很有价值。

### 写在最后

这篇文章里的每一个项目，都是从一个想法开始，在几十个小时内变成了一个能跑起来的产品。有些在比赛结束后已经不再更新了，有些还在持续迭代，有些甚至已经有了真实用户。

我们做黑客松赞助这件事，最初的想法很简单：让好的模型能力到达更多开发者手里，看看他们会用它做出什么来。现在看来，他们做出来的东西，比我们预期的更有意思。
